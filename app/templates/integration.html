Integration Guide / 集成指南

        

          
1. System Overview / 系统概览

          
Mingdao is the master employee database. ADMS receives employee data from Mingdao through REST APIs, stores it in device_user, prepares per-device records in device_user_sync, and later communicates with attendance devices through ADMS GETREQUEST. Administrators must not edit ADMS database records directly.

          
Mingdao
  -> REST API
  -> ADMS Server
  -> Attendance Device


          
2. Authentication / 鉴权

          
Every Mingdao API request must include the current API token. The token is configured as MINGDAO_API_TOKEN in .env.

          
Authorization: Bearer <API_TOKEN>
X-API-Key: <API_TOKEN>


          
3. Available APIs / 当前可用接口

          

            
APIMethodPurpose

            
              
/api/v1/usersPOSTCreate or update one Mingdao employee in ADMS.

              
/api/v1/users/batchPOSTCreate or update up to 500 Mingdao employees in one request.

              
/api/v1/usersGETList current Mingdao-sourced users.

              
/api/v1/users/{employee_id}GETGet one Mingdao-sourced user by employee ID.

            
          


          
4. Mingdao Configuration Guide / 明道配置说明

          
Configure Mingdao HTTP request nodes with the API URL shown on this page. Use method POST, header Content-Type: application/json, and either Authorization: Bearer <API_TOKEN> or X-API-Key: <API_TOKEN>.

          
{
  "employee_record_id": "659823456789",
  "employee_id": "ESM0001",
  "pin": "1",
  "name": "Raymon",
  "department": "IT",
  "card_no": "",
  "privilege": 0,
  "enabled": true
}

          
Successful response:

          
{
  "success": true,
  "employee_id": "ESM0001",
  "employee_record_id": "659823456789",
  "pin": "1",
  "message": "User synchronized successfully.",
  "changed": true,
  "sync_records": 1
}

          
Error response examples include 401 Unauthorized for missing or invalid token, 413 when batch size exceeds 500 users, and 422 for invalid JSON, missing employee_record_id, or unsupported privilege.


          
5. Synchronization Flow / 同步流程

          
Mingdao
  -> REST API
  -> device_user
  -> device_user_sync
  -> Attendance Device

          
When a user is created or changed, ADMS creates or updates one PENDING sync record per registered device. Online devices receive user commands through ADMS GETREQUEST. Offline devices remain PENDING and can be synchronized after reconnect.


          
6. Troubleshooting / 故障排查

          
    
            
  • 401 Unauthorized: Token is missing, default, or invalid.
    • 
            
  • 404 API Not Found: Confirm the path starts with /api/v1.
    • 
            
  • API Timeout: Confirm ADMS server port 4370 is reachable.
    • 
            
  • Device Offline: User sync remains PENDING until the device reconnects.
    • 
            
  • No Online Device: Check Operation Monitor and device heartbeat.
    • 
            
  • No API Request Received: Verify Mingdao request URL, headers, and network access.
    •

Attendance Sync Guide / 考勤同步说明

        

          
1. Purpose / 用途

          
This page stores Mingdao OpenAPI configuration and can manually upload saved ADMS attendance events to Mingdao. The local sync table prevents the same ADMS attendance event from being uploaded twice.


          
2. Mingdao OpenAPI Authentication / 明道 OpenAPI 鉴权

          
The connection test uses Mingdao OpenAPI headers copied from Mingdao: HAP-Appkey and HAP-Sign.

          
HAP-Appkey: <AppKey>
HAP-Sign: <Sign>


          
3. Worksheet and Field Mapping / 工作表与字段映射

          
Configure the Mingdao Attendance Worksheet ID and target field IDs for Employee Record ID, Check Time, Device Name, and Device SN. The ADMS source fields are fixed; only Mingdao target field IDs are editable.


          
4. Duplicate Prevention / 防重复上传

          
DT014 uses attendance_event.id as the local idempotency key through attendance_mingdao_sync. Records with SYNCED status are never uploaded again. New records sync automatically; records missing employee_record_id remain pending without calling Mingdao, and failed network/API writes retry after the configured interval in minutes.
